# S6 V15B fresh-basis back-plan confirmation

V15B is the one confirmation authorized by the sealed V15A positive aggregate
evidence. It repeats the exact alpha-zero, five-panel train-only diagnostic on
another deterministic 32-direction basis. No validation, OOD, heldout, or
benchmark outcome may be opened, and no model update may be applied.

## Why the run remains paired

The middle-late 20--23 arm is a non-promotable nuisance control, not a second
candidate. Running it on the same new basis prevents a basis draw from being
mistaken for a back-layer effect. Only back layers 36--39 can be confirmed.
Both arms have the same frozen capacity and use identical panels, generation,
objective, direction order, and basis. Execution order is middle-late then
back; sorted persisted JSON is replayed according to this explicit arm order.

## Conjunctive confirmation gate

Every median and worst endpoint must pass every applicable rule:

1. Back cosine endpoints must be at least V13 plus `0.05`; back sign endpoints
   must be no lower than V13.
2. Back cosine endpoints must be at least the same-basis middle-late control
   plus `0.05`; back sign endpoints must be no lower than the control.
3. Back cosine endpoints must be no more than `0.05` below the V15A back
   endpoint. This reuses the already-preregistered material-effect unit.
4. Back sign endpoints must be no more than `1/32` below the V15A back
   endpoint. This is one direction-coordinate of the frozen population.
5. Both arms must have nonzero panel spreads and every restoration, base-probe,
   population-boundary, dense-hash, hardware, fresh-basis, and alpha-zero
   integrity audit must pass.

The tolerances and all endpoints are fixed before runtime implementation; no
post-hoc tolerance or endpoint selection is allowed. Failure retains V13 and
keeps all evaluation surfaces closed. Passing authorizes only a separate
preregistration for a nonzero-alpha back-plan train-update experiment. It does
not itself authorize an update or evaluation.

The preregistration commit does not authorize a GPU launch. Launch requires a
second committed, fail-closed runtime bundle with exact dry-run hashes.
